# AI 小说助手 — 技术方案说明

> 供外部 AI 交叉审计，验证技术方案合理性。

---

## 一、目标

用 AI 自动生成多章长篇小说，每次跑 10 章，**核心诉求是一致性，不追求文学质量**。

具体要保证（按优先级排序）：

| 优先级 | 一致性维度 | 定义 |
|--------|----------|------|
| P0 | 物品归属 | 某物品属于谁，不能在不同角色间漂移 |
| P0 | 角色关系 | 双向关系一致（A 的师父是 B → B 的徒弟必有 A） |
| P1 | 角色状态 | 位置、等级、健康状态不矛盾 |
| P1 | 目标跟踪 | 角色的当前目标不遗漏、不已完成仍标记活跃 |
| P2 | 伏笔回收 | 埋下的伏笔能在合理章节被检索到并回收 |
| P2 | 情节连贯 | 前文发生的事件后续章节不遗忘不矛盾 |

## 二、技术选型

| 组件 | 干什么 | 用什么 | 成本 |
|------|--------|--------|------|
| 正文生成 LLM | 写每一章小说 | DeepSeek deepseek-chat（temperature 0.7） | ~0.015 元/章 |
| 分析提取 LLM | 从章节中提取结构变化 | DeepSeek deepseek-chat（temperature 0.3） | ~0.001 元/章 |
| 审计校验 LLM | 每 5 章交叉验证状态一致性 | DeepSeek deepseek-chat（temperature 0.1） | ~0.001 元/5章 |
| Embedding | 文本→1024 维向量 | 阿里云千问 text-embedding-v3 | 极低 |
| 向量存储 | 存储并检索相关前文 | 自制：JSON 文件 + 余弦相似度暴力搜索 | 免费 |
| 结构化存储 | 角色状态/时间线/摘要/伏笔 | JSON 文件 | 免费 |

**为什么不用专用向量数据库？** 100 章约 400 个文本块、400 个 1024 维向量，暴力搜索只需 ~400 次余弦计算，远不到需要 Pinecone/Milvus 的量级。JSON 零运维。

## 三、核心数据结构

### 3.1 角色状态 `state/characters.json`

每个角色是一个键，值包含以下字段：

```
字段            类型      注入 prompt 时？   有界？
────────────────────────────────────────────────
location        string    全量（一行）       N/A
health          string    非"健康"时显示     N/A
level           string    有值则显示          N/A
status          string    全量               N/A
inventory       string[]  截断至最近10件      上限15件（存储）
goals           string[]  截断至最近5个       上限8个（存储）
relationships   object    全量               N/A
last_seen_ch    number    全量               N/A
```

**关键设计**：
- `goals` 和 `inventory` 在 prompt 注入时做截断，防止无限膨胀撑爆上下文窗口
- 每章分析时要求 AI 输出**完整清单**（非增量），防止漂移
- 每 10 章做一次硬截断清理（防止 AI 分析没删干净）
- `relationships` 需保持双向一致，依赖审计步骤修正

### 3.2 事件时间线 `state/timeline.json`

```
[{ "chapter": 3, "event": "林青云突破练气六层", "timestamp": 1720000000000 }]
```

- 时间线是逐个事件追加的，永不修改
- 注入 prompt 时取最近 8 条（防膨胀）
- 总量超过 200 条时自动截断保留最近 150 条
- `timestamp` 目前是存储时间（Date.now），不是故事内时序

### 3.3 章节摘要 `state/summaries.json`

```
[{ "chapter": 5, "summary": "林青云在宗门大比中获胜..." }]
```

- 注入 prompt 时取最近 5 章
- 每章约 100-200 字

### 3.4 伏笔 `state/foreshadows.json`

```
[{ "chapter": 3, "description": "后山禁地深处有秘密洞穴", "characters": ["林青云", "苏小月"] }]
```

- 存储但不直接注入 prompt
- 依赖向量检索召回（伏笔文本被 embed 进向量库）

### 3.5 向量库 `vector_index/`

- `chunks.json`：正文按 600 字/块分块，含 meta（章节号、标题）
- `vectors.json`：每块对应的 1024 维向量
- 检索方式：**多层检索** — 用章节大纲和活跃角色名作为两个独立 query，分别取 top-3，去重合并为 top-6

## 四、单章生成流程

```
Step 0: 检索层
  ├── query_1 = 本章大纲
  ├── query_2 = "林青云、苏小月 的相关情节"  （活跃角色名拼接）
  ├── 各自 top-3，去重合并 top-6
  └── 结果作为 system message 注入

Step 1: 组装 Prompt（约 2000-3000 token）
  ├── 世界观（静态，始终注入）
  ├── 角色当前状态（bounded，见 3.1）
  ├── 角色关系
  ├── 近期事件（最近 8 条）
  ├── 前文摘要（最近 5 章）
  ├── 文风要求
  ├── 本章大纲
  └── 格式要求（强制"第N章 标题"开头）

Step 2: AI 生成正文
  └── DeepSeek, temperature 0.7, maxTokens 4096

Step 3: 后处理
  ├── fixChapterHeader()：修正章节标题
  └── 保存章节文件

Step 4: 分析 & 更新（afterWrite）
  ├── AI CoT 分析：给定上一章状态 + 本章正文 → 先推理再 JSON
  │   ├── 提取 summary（章节摘要）
  │   ├── 提取 events（关键事件）
  │   ├── 提取 foreshadows（新埋伏笔）
  │   └── 提取 characterChanges（角色状态变化，完整清单非增量）
  │
  ├── validateAnalysis()：结构校验
  │   ├── 角色名是否在 characters.json 中存在
  │   ├── inventory/goals 是否为数组
  │   ├── 摘要长度是否合理
  │   └── 异常字段忽略并告警
  │
  ├── 写入 memory（不保存）
  │   ├── addSummary()
  │   ├── addEvent() × N
  │   ├── addForeshadow() × N
  │   └── updateCharacters()
  │
  ├── 正文分块→embedding→向量库
  │
  ├── memory.save() + vectorStore.save()  ← 统一写盘
  │
  └── 周期审计
      ├── 每 5 章: stateAudit()  ← 交叉校验（见第五节）
      └── 每 10 章: cleanupState()  ← 硬截断
```

## 五、一致性保障：三层防线

### 第一层：分析校验（每章）

`validateAnalysis()` 在 AI 分析输出 JSON 后立即执行**结构级**检查，不涉及额外 API 调用：
- 角色名是否存在于 characters.json
- inventory/goals 是否真的是数组
- 摘要长度是否 ≥ 10 字

作用：拦截 AI 偶尔输出的非数组字段（如 goals 是字符串"无变化"），防止 `join()` 报错或状态损坏。

**不检查语义正确性**（无法在不增加 API 调用的情况下做到）。

### 第二层：交叉审计（每 5 章）

`stateAudit()` 是**独立的 AI 调用**，与 per-chapter 分析不同：

```
输入（给审计 LLM）：
  - 最近 8 章摘要
  - 最近 40 条事件时间线
  - 完整的 characters.json

检查项：
  1. 物品归属是否与摘要矛盾？
  2. 等级/能力是否与摘要匹配？
  3. 目标是否已完成但仍标记活跃？
  4. 双向关系是否一致？
  5. 是否有明显不合理字段（死人标活跃等）？

输出：需要修正的字段（diff），不是完整状态
```

**为什么是每 5 章而不是每章？**
- 每章审计成本高且没必要（一篇文章很难产生严重累积误差）
- 5 章约 8000 字，足够产生可检测的漂移（物品归属、关系偏差）
- 间隔太短会重复修正同样的问题

### 第三层：硬截断（每 10 章）

`cleanupState()` 是纯代码逻辑（无 AI 调用）：
- inventory 超过 15 件 → 截断保留最近 15 件
- goals 超过 8 个 → 截断保留最近 8 个
- 时间线超过 200 条 → 截断保留最近 150 条

作用：兜底，防止审计 AI 没删干净导致存储文件物理膨胀。

## 六、已知问题 / 风险

| # | 问题 | 严重度 | 状态 | 说明 |
|---|------|--------|------|------|
| 1 | AI 分析可能错提、漏提状态变化 | 中 | 已有缓解 | 第一层校验 + 第二层审计可纠正大部分 |
| 2 | 审计 LLM 和 分析 LLM 用的同一个模型 | 低 | 已知 | DeepSeek 是唯一 LLM，无法用独立模型交叉验证 |
| 3 | 文本分块按字数硬切 | 低 | 已知 | 600 字/块，可能在场景中间切断，影响检索精度 |
| 4 | 向量检索不保证召回伏笔 | 低 | 已知 | 依赖语义相似，可能漏掉不相关的伏笔 |
| 5 | 时间线 `timestamp` 是存储时间 | 低 | 已知 | 非故事内时间，不影响一致性 |
| 6 | 无章节重写/回滚机制 | 低 | 已知 | 某章生成质量差无法撤销对 state 的影响 |
| 7 | 大纲不可跳过/插入 | 低 | 已知 | 章节编号完全依赖 outline 数组索引 |

## 七、项目文件结构

```
novel-assistant/
├── run.js              # 主入口：检测已有章节，编排逐章生成
├── story-config.js     # 故事设定（世界观/角色/大纲）
├── lib/
│   ├── llm.js          # API 封装（DeepSeek 对话 + 千问 Embedding）
│   ├── writer.js       # 写作引擎（prompt 组装/生成/分析/校验/审计）
│   ├── memory.js       # 结构化记忆（角色/时间线/摘要/伏笔）
│   └── vector-store.js # 向量库（JSON + 余弦相似度 + 多层检索）
├── chapters/           # 生成的小说正文 .txt
├── state/              # characters.json / timeline.json / summaries.json / foreshadows.json
└── vector_index/       # chunks.json / vectors.json
```
